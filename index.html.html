The case for this setup

Why, when, and how this actually helps

Why

Agents forget. Files don't.

Every new chat with an AI agent starts blank. Without written memory, you re-explain your architecture, your naming conventions and yesterday's decisions every single time.

  • Decisions survive across sessions and even across people
  • New teammates (human or agent) get up to speed by reading, not asking
  • Nothing important lives only inside a chat log
When

Multi-module, multi-week projects

This is overkill for a weekend script. It pays off the moment a project has more than one moving part and more than one working session.

  • A backend, a frontend and a database evolving together
  • Work that spans days or weeks, not one sitting
  • More than one person — or more than one agent — touching the code
How

Three lanes, one shared bus

Each folder gets its own agent that only knows its own domain deeply — plus a small shared layer at the root so no lane drifts out of sync with the other two.

  • Backend agent never has to reason about Angular templates
  • Frontend agent never has to reason about SQL migrations
  • A thin root layer keeps all three pointed at the same goal
For anyone, technical or not

What "context engineering" actually means

Picture hiring three specialist contractors for a house: an electrician, a plumber and a structural engineer. If none of them ever left notes, every single morning they'd have to re-inspect the whole house before doing any work — and they'd keep contradicting each other.

Context engineering is simply making each contractor keep three notebooks — what they've learned, what they're planning next, and exactly where they left off — plus one shared clipboard by the front door that all three check before starting.

memory.md — "What I've learned"

Standing decisions, conventions and lessons that shouldn't be re-learned. Rarely changes; grows slowly.

plan.md — "What I'm planning"

The current roadmap and priorities for this folder only. Updated when priorities shift.

state.md — "Where I left off"

A snapshot of progress right now — done, in-progress, blocked. Updated at the end of every session.

Context engineering in action

How the three agents stay in sync

Each agent reads its own three files before doing anything. When a decision affects more than one lane, it gets written up to the shared bus so the other two agents pick it up on their next session.

Backend agent

spring-boot-backend/.context/
memory.md API conventions, entity rules
plan.md this sprint's endpoints
state.md last endpoint finished

Frontend agent

angular-frontend/.context/
memory.md component & style rules
plan.md this sprint's screens
state.md last screen finished

Database agent

oracle-database/.context/
memory.md naming, indexing rules
plan.md this sprint's migrations
state.md last migration applied

Shared context bus — parent-project/shared-context/

Cross-cutting decisions (a renamed field, a new API contract, a changed table) travel here so all three agents see the same truth.

The loop every session follows
01
Read context

Agent opens its AGENTS.md, memory.md, plan.md and state.md — plus the shared-context files.

02
Do the work

Code, migrate, or design — scoped strictly to that agent's own folder.

03
Update state.md

What got done, what's next, what's blocked — written before the session ends.

04
Promote to shared

Anything the other two agents need to know gets copied up to shared-context/.

Step by step, no jargon assumed

Building this from an empty folder

Each step below is something you can literally do in order. Tap a step to expand it.

1

Install OpenCode

+

OpenCode is a command-line AI coding assistant, similar in spirit to a chat app but running from your terminal, inside your project folder.

curl -fsSL https://opencode.ai/install | bash opencode --version

If that's your first time in a terminal: a terminal is just a text window where you type commands instead of clicking icons. You'll only need a handful of commands for this whole project.

2

Create the parent folder and three subfolders

+

If your Spring Boot, Angular and Oracle folders already exist somewhere, move or copy them so they sit side by side inside one parent folder.

mkdir parent-project cd parent-project mkdir spring-boot-backend angular-frontend oracle-database

Nothing inside those three folders needs to change — the context files sit alongside your existing code, they don't replace anything.

3

Write the root AGENTS.md

+

This is the one file every agent reads first, no matter which folder it's working in. Keep it short: what the project is, the three modules, and the one rule that matters most — stay inside your own lane, promote shared decisions to shared-context/.

You can generate a first draft automatically:

opencode > /init

The /init command scans your codebase and writes a starting AGENTS.md for you, which you then edit by hand.

4

Define the three agents in opencode.json

+

An "agent" in OpenCode is a named persona: a system prompt plus which folder it should focus on. Registering three of them means you can switch between "the Spring Boot expert" and "the Angular expert" without re-explaining who they are.

{ "$schema": "https://opencode.ai/config.json", "default_agent": "backend", "agent": { "backend": { "description": "Spring Boot specialist" }, "frontend": { "description": "Angular 19 specialist" }, "database": { "description": "Oracle SQL specialist" } } }

See the Templates section below for the full persona files that go with each agent.

5

Create the memory / plan / state files

+

Inside each of the three project folders, add a .context folder with three markdown files. Copy the starter templates from the Templates section — they're intentionally short so you'll actually keep them updated.

mkdir -p spring-boot-backend/.context mkdir -p angular-frontend/.context mkdir -p oracle-database/.context
6

Create the shared-context layer

+

At the parent level, create one more shared-context folder with its own memory/plan/state trio — this is the "clipboard by the front door" every agent checks.

mkdir shared-context touch shared-context/memory.md shared-context/plan.md shared-context/state.md
7

Start a session with the right agent

+

Launch OpenCode from the parent folder and switch to the agent for the module you're working on (the Tab key cycles between agents in the terminal UI).

cd parent-project opencode > Tab (cycle to "frontend" agent) > Read angular-frontend/.context and continue where we left off
8

End every session by updating state.md

+

This is the single habit that makes the whole system work. Before closing a session, ask the agent to update its own state.md with what was finished, what's next, and anything blocked.

> Update .context/state.md with today's progress and next steps

If a decision affects the other two modules, ask the agent to also add a line to shared-context/state.md or memory.md.

9

Commit the context files to git

+

Treat the memory/plan/state files like documentation, not scratch notes — commit them alongside code changes so the history of decisions is preserved and reviewable, just like the code itself.

git add .context shared-context AGENTS.md git commit -m "docs(context): update backend state after auth endpoints"
10

Review and prune weekly

+

memory.md tends to grow. Once a week, skim each one and delete anything that's no longer true or no longer useful — stale context is worse than no context, because an agent will trust it.

Copy, paste, adjust

Starter templates

These are deliberately short. A context file you won't update is worse than no context file at all.

spring-boot-backend/.context/memory.md
# Backend Memory

## Standing decisions
- All REST endpoints are versioned under /api/v1
- DTOs are separate from JPA entities, always
- Errors return a consistent { code, message } shape

## Conventions
- Package by feature, not by layer (e.g. `order/`, not `controllers/`)
- Every entity maps to one Oracle table owned by the database agent

## Lessons learned
- (add as they happen — one line each, dated)

spring-boot-backend/AGENTS.md
# Spring Boot Backend — Agent Instructions

You are the backend specialist for this repository. Work only inside
`spring-boot-backend/` unless explicitly asked to cross into another module.

## Before doing anything
1. Read `.context/memory.md`, `.context/plan.md`, `.context/state.md`
2. Read `../shared-context/memory.md` and `../shared-context/state.md`

## Stack
Java 21, Spring Boot 3.x, Maven, JUnit 5, Oracle JDBC

## After finishing work
Update `.context/state.md`. If the change affects the frontend or
database schema, add a note to `../shared-context/state.md`.
